# Assest — Request & Data Flow (ingest → store → retrieve → reason → respond)

This document maps the runtime flows for requests and data through the system.

1) Ingestion (external sources → system)
- Connectors: `backend/connectors/` (Google Drive, Notion, Slack, etc.) fetch raw documents.
- Ingest tasks run via `backend/ingestion/` and worker pool under `backend/workers/`.
- Documents are normalized and persisted into vector store (Qdrant) and relational DB for metadata.

2) Storage
- Vector embeddings: generated by embedding model (`backend/core/config.py` controls provider) and stored in Qdrant (`data/qdrant` or external host).
- Metadata: `backend/models/` SQLAlchemy models persisted in SQLite/Postgres.

3) Retrieval
- Query path: `backend/api/query` receives queries, `backend/query/query_service.py` orchestrates retrieval.
- `SemanticCache` sits in `backend/query/semantic_cache.py` to short-circuit frequent queries.
- Retrieval uses `backend/retrieval/` helpers to fetch vectors from Qdrant and rank candidates.

4) Reasoning / Orchestration
- `backend/reasoning/ReasoningOrchestrator` (LangGraph orchestration) coordinates multi-agent planning, analysis, and synthesis.
- Agents live in `backend/reasoning/agents/` (planner, analyst, synthesizer); they call into `backend/core/llm_impl.SharedLLMClient` for generation.
- `SharedLLMClient` centralizes provider selection (OpenRouter/Groq/Proxy/local) and applies fallbacks/retries.

5) Streaming & Response
- Streaming endpoint: `POST /api/query/stream` implemented in `backend/api/query` calls `QueryService.stream_query()` which yields SSE events.
- SSE event types: `conversation`, `status`, `token`, `sources`, `done` (structured JSON under `data: ...`).
- Frontend consumes the stream in `web/src/app/chat/[id]/page.tsx`, parsing `status` and `token` events to update UI progressively.

6) Observability & Tracing
- LLM calls instrumented via Langfuse (when available) in `backend/core/llm_impl.py` and logged via standard logging.
- Add a `call_id` per LLM invocation to correlate logs and SSE events back to the originating query.

Notes / Recommendations
- Ensure `call_id` flows from `query_service` → `ReasoningOrchestrator` → `SharedLLMClient` so traces can be correlated end-to-end.
- Keep SSE contract stable: event JSON must include `type` and `timestamp` and optionally `meta.request_id`.
